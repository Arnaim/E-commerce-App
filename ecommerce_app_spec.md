# E-Commerce Mobile App — Project Specification

## Overview

Build a **mobile-only e-commerce application** where users can buy and sell products securely. The app must prioritize data security, payment safety, and scalability.

---

## Tech Stack

| Layer | Choice |
|---|---|
| Framework | Flutter (cross-platform, iOS + Android) |
| Backend & Database | Firebase (Firestore, Auth, Storage, Cloud Functions) |
| Payments | Stripe (via `flutter_stripe` + Firebase Cloud Functions) |
| State Management | Provider |
| Navigation | go_router |

## Features

### 1. User Authentication

- Sign up with email and password using Firebase Auth
- Log in and log out
- Auth state persists across app restarts
- Role field in Firestore: `buyer` or `seller` (set at sign-up)
- Route guards: unauthenticated users redirected to login

**Firestore: `users/{uid}`**
```json
{
  "uid": "string",
  "email": "string",
  "displayName": "string",
  "role": "buyer | seller",
  "createdAt": "timestamp"
}
```

---

### 2. Product Listing & Search

- Home screen shows all active product listings
- Products pulled from Firestore `products` collection
- Search bar filters by product name (client-side on fetched results)
- Category filter chips (e.g. Electronics, Clothing, Books)
- Product card shows: image, name, price, seller name
- Tap product → Product Detail screen (full description, images, add to cart)

**Firestore: `products/{productId}`**
```json
{
  "id": "string",
  "sellerId": "string",
  "sellerName": "string",
  "name": "string",
  "description": "string",
  "price": "number",
  "category": "string",
  "imageUrls": ["string"],
  "stock": "number",
  "createdAt": "timestamp",
  "isActive": "boolean"
}
```

### 3. Cart (local state, no Firestore)

- Users can add/remove products from cart
- Cart managed in Provider (in-memory, per session)
- Cart screen shows items, quantities, subtotal
- Quantity can be adjusted per item
- "Proceed to Checkout" button

### 4. Secure Payment Gateway (Stripe)

**Flow:**

1. User taps "Pay Now" on Checkout screen
2. Flutter calls a Firebase Cloud Function (`createPaymentIntent`) with amount + currency
3. Cloud Function uses Stripe secret key to create a PaymentIntent and returns `clientSecret`
4. Flutter uses `flutter_stripe` to show the native Stripe payment sheet
5. On success, order is written to Firestore and cart is cleared

**Firebase Cloud Function (`functions/index.js`):**
```javascript
const functions = require('firebase-functions');
const stripe = require('stripe')(process.env.STRIPE_SECRET_KEY);

exports.createPaymentIntent = functions.https.onCall(async (data, context) => {
  if (!context.auth) throw new functions.https.HttpsError('unauthenticated', 'Login required');
  
  const { amount, currency } = data;

  const paymentIntent = await stripe.paymentIntents.create({
    amount: Math.round(amount * 100), // cents
    currency: currency || 'usd',
  });

  return { clientSecret: paymentIntent.client_secret };
});
```

**Flutter payment trigger (`payment_service.dart`):**
```dart
Future<void> makePayment(double amount) async {
  final callable = FirebaseFunctions.instance.httpsCallable('createPaymentIntent');
  final result = await callable.call({'amount': amount, 'currency': 'usd'});
  final clientSecret = result.data['clientSecret'];

  await Stripe.instance.initPaymentSheet(
    paymentSheetParameters: SetupPaymentSheetParameters(
      paymentIntentClientSecret: clientSecret,
      merchantDisplayName: 'My Shop',
    ),
  );

  await Stripe.instance.presentPaymentSheet();
}
```

> **Security rule:** The Stripe secret key is stored only as a Firebase secret (`STRIPE_SECRET_KEY`) and accessed only in Cloud Functions. It is never present in Flutter code.

### 5. Order History & Tracking

- After successful payment, an order document is written to Firestore
- Buyers can view all their past orders in "My Orders" screen
- Each order shows: items, total, date, status
- Order statuses: `pending` → `processing` → `shipped` → `delivered`

**Firestore: `orders/{orderId}`**
```json
{
  "id": "string",
  "buyerId": "string",
  "sellerId": "string",
  "items": [
    { "productId": "string", "name": "string", "price": "number", "quantity": "number" }
  ],
  "total": "number",
  "status": "pending | processing | shipped | delivered",
  "createdAt": "timestamp",
  "stripePaymentIntentId": "string"
}
```

### 6. Seller Dashboard

- Only visible to users with `role == "seller"`
- Shows list of seller's own products with stock count and status
- "Add Product" button → form to upload images + fill product details
- Images uploaded to Firebase Storage, download URLs saved to Firestore
- Seller can toggle product active/inactive
- Seller can view incoming orders for their products and update order status

## Firestore Security Rules

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    match /users/{uid} {
      allow read, write: if request.auth.uid == uid;
    }

    match /products/{productId} {
      allow read: if true;
      allow create, update, delete: if request.auth != null
        && get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role == 'seller'
        && resource.data.sellerId == request.auth.uid;
    }

    match /orders/{orderId} {
      allow read: if request.auth != null
        && (resource.data.buyerId == request.auth.uid
            || resource.data.sellerId == request.auth.uid);
      allow create: if request.auth != null;
      allow update: if request.auth != null
        && get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role == 'seller'
        && resource.data.sellerId == request.auth.uid;
    }
  }
}
```

## Firebase Storage Rules

```
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    match /products/{sellerId}/{allPaths=**} {
      allow read: if true;
      allow write: if request.auth != null && request.auth.uid == sellerId;
    }
  }
}
```

## Non-Functional Requirements

- All Firestore reads/writes must go through service classes, not directly from UI
- No business logic in widgets; use Providers
- Stripe secret key must never appear in Flutter source code
- Images must be compressed before upload (use `image_picker` with `maxWidth: 1024`)
- App must handle network errors gracefully with user-facing error messages
- Use Stripe test mode during development; switch to live keys only for production release
